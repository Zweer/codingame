# Puzzle Solving Guidelines

## Directory Structure

Puzzles live under `src/<level>/<prettyId>/` where level directories are numbered for sorting:

| Directory | Level |
|-----------|-------|
| `00-tutorial` | tutorial |
| `01-easy` | easy |
| `02-medium` | medium |
| `03-hard` | hard |
| `04-expert` | expert |
| `05-optim` | optim |
| `06-multi` | multi |
| `07-codegolf-easy` | codegolf-easy |
| `08-codegolf-medium` | codegolf-medium |
| `09-codegolf-hard` | codegolf-hard |
| `10-codegolf-expert` | codegolf-expert |

## File Structure per Puzzle

Each puzzle directory should contain:

```
src/<level>/<prettyId>/
├── handler.ts          # Pure logic: exported function(s), no I/O
├── index.ts            # CodinGame glue: readline → handler → console.log
├── handler.test.ts     # Tests for handler (optional, when test cases available)
├── main.rs             # Rust solution (optional, for performance-critical puzzles)
└── gemini-reasoning.md # AI reasoning (auto-generated by solve script)
```

### handler.ts
- Exports one or more pure functions
- Takes parsed input as parameters, returns the answer
- NO `readline()`, NO `console.log()`, NO side effects
- This is the testable core of the solution

### index.ts
- Thin glue layer: reads input via `readline()`, calls handler, prints output
- Uses `// @ts-expect-error` for the CodinGame `readline` global
- Should be as minimal as possible

### Example

```typescript
// handler.ts
export function solve(temperatures: number[]): number {
  if (temperatures.length === 0) return 0;
  return temperatures.reduce((closest, t) =>
    Math.abs(t) < Math.abs(closest) || (Math.abs(t) === Math.abs(closest) && t > 0) ? t : closest
  );
}

// index.ts
import { solve } from './handler.js';
const n = Number.parseInt(readline());
const temps = readline().split(' ').map(Number).slice(0, n);
console.log(solve(temps));
```

## Language Choice

### TypeScript (default)
- Use for all puzzles unless performance is insufficient
- Prefer clean, structured code: classes, interfaces, proper types
- Use standard data structures (Map, Set, arrays)

### Rust (performance-critical)
- Use when TypeScript solution exceeds time limits (typically hard/expert puzzles)
- Write as standalone `main.rs` file (no Cargo project needed — CodinGame compiles single files)
- Keep the TypeScript solution too when possible (for readability/testing)

### When to switch to Rust
- Puzzle has strict time limits (< 1s for large inputs)
- Algorithm is already optimal but TS is too slow (e.g., heavy graph traversal, large matrix ops)
- The puzzle explicitly requires low-level performance

## Solving Approach

### With Kiro (interactive, local)
- Discuss the puzzle, understand the problem
- Plan the algorithm together
- Implement in handler.ts pattern
- Test locally with vitest if test cases are available

### With Gemini (automated, via API)
- Used by `npm run script:solve` for batch solving
- Generates `index.ts` + `gemini-reasoning.md`
- Solutions may not follow handler pattern (acceptable for auto-generated code)
- Manual review recommended before submitting to CodinGame

## Data Source

- `data/puzzles.json` — local database of all CodinGame puzzles (fetched via API)
- `npm run script:retrieve` — refreshes the puzzle list from CodinGame
- `npm run script:readme` — updates README checkboxes based on which `src/<level>/<prettyId>/` directories exist
- Puzzle statements are fetched on-demand via `findProgressByPrettyId` API call

## Submitting Solutions

There is NO API to submit solutions to CodinGame puzzles.
Solutions must be manually copy-pasted into the CodinGame web IDE.
